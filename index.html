Bootstrapping — Establecer confianza

El cliente descarga el certificado raíz de la CA y valida su autenticidad usando un fingerprint seguro compartido fuera de banda.

Cliente
→
step ca bootstrap
→
CA raíz
Comandostep ca bootstrap --fingerprint <hash>
ResultadoCert. raíz almacenado localmente
ValidaFingerprint SHA-256 del cert. raíz
¿Por qué?Permite confiar en futuras conexiones TLS
PKIRoot CATrust anchorFingerprint
shell · bootstrap 
# 1. Obtener el fingerprint del server (ejecutar en el servidor CA)
step certificate fingerprint $(step path)/certs/root_ca.crt

# 2. Bootstrap en el cliente (usar el fingerprint obtenido arriba)
step ca bootstrap \
  --ca-url https://ca.internal:9000 \
  --fingerprint abc123...sha256hash

# Resultado: descarga root_ca.crt y configura $STEPPATH/config/defaults.json
# Verificar que quedó bien:
step ca health

shell · autenticación por provisioner 
# JWK: generar token JWT (el más común para agentes IA)
step ca token agent.service.internal \
  --provisioner jwk-agent \
  --key /etc/step/jwk-key.json

# OIDC: autenticación interactiva con Google / GitHub
step ca token agent.service.internal \
  --provisioner google-oidc

# Cloud AWS: usa el identity document de la instancia EC2
step ca token agent.service.internal \
  --provisioner aws

# K8s: usa el Service Account token del pod
step ca token agent.service.internal \
  --provisioner k8ssa \
  --k8s-token-file /var/run/secrets/kubernetes.io/serviceaccount/token

shell · claves y csr 
# Generar solo el par de claves (sin CSR todavía)
step crypto keypair \
  agent.pub agent.key \
  --kty EC \
  --curve P-256

# Crear CSR a partir del par de claves existente
step certificate create \
  agent.service.internal agent.csr agent.key \
  --csr \
  --san agent.service.internal \
  --san 127.0.0.1

# Inspeccionar el CSR generado
step certificate inspect agent.csr

Firma y emisión

El cliente envía CSR + token al endpoint /1.0/sign. La CA verifica, firma con su clave intermedia y devuelve el certificado.

CSR + Token
→
POST /1.0/sign
→
CA verifica
↓
✓ Certificado firmado (24h)
Endpoint/1.0/sign
Firma conClave intermedia de la CA
Duración por defecto24 horas
MotivoShort-lived = menor superficie de ataque
Intermediate CAX.509Short-lived
shell · solicitar y firmar certificado 
# Flujo completo: token + certificado en un solo comando (recomendado)
step ca certificate agent.service.internal \
  agent.crt agent.key \
  --provisioner jwk-agent \
  --not-after 24h \
  --san agent.service.internal

# Con token pre-generado
TOKEN=$(step ca token agent.service.internal --provisioner jwk-agent)
step ca certificate agent.service.internal \
  agent.crt agent.key \
  --token $TOKEN

# Firmar un CSR existente (si generaste el CSR por separado)
step ca sign agent.csr agent.crt \
  --token $TOKEN

# Inspeccionar el certificado emitido
step certificate inspect agent.crt

Renovación automatizada

Un proceso en segundo plano monitorea el tiempo y solicita automáticamente un nuevo certificado antes de que expire el actual.

step ca renew --daemon
→
Monitorea tiempo
↓
~66% del tiempo transcurrido
→
Solicita renovación
↓
♻ Nuevo cert. (24h más)
ProcesoDaemon en background
Cuándo renuevaAntes del 66% de vida útil
Sin intervenciónTotalmente automático
Fallo = expiraciónSi se bloquea la renovación
DaemonAuto-renewalZero downtime
shell · renovación automática 
# Modo daemon: renueva automáticamente (recomendado para producción)
step ca renew \
  agent.crt agent.key \
  --daemon \
  --exec "systemctl reload my-agent"
# --exec corre ese comando después de cada renovación exitosa

# Renovación manual (one-shot, útil para testing)
step ca renew agent.crt agent.key --force

# Ver cuándo expira el cert actual
step certificate inspect agent.crt --format json | jq .validity

# Systemd unit: crear /etc/systemd/system/step-renew.service
# ExecStart=/usr/bin/step ca renew /etc/step/agent.crt /etc/step/agent.key \

Revocación

Dos enfoques según la urgencia. Seleccioná el tipo para ver el flujo detallado.

Cert. activo (24h)
→
Tiempo llega a 0
→
Expirado
↓
CA bloquea renovación
→
Acceso denegado
El método estándar. Con certs de 24h, comprometer uno da ventana mínima. El servidor simplemente deja de renovar si el cert está comprometido.
shell · revocación pasiva 
# Bloquear renovación de un certificado específico
# (pasiva: el cert expira solo, solo bloqueamos la renovación futura)
step ca revoke \
  --serial $(step certificate inspect agent.crt --format json | jq -r .serialNumber) \
  --reason "Credential rotated" \
  --reasonCode 4
# reasonCode 4 = superseded

# El cert sigue válido hasta expirar (en ≤24h)
# La CA rechaza el siguiente intento de renovación automáticamente
Admin: step ca revoke
→
CA añade nro. serie a CRL
↓
Cliente consulta /1.0/crl
→
Compara nro. serie
↓
🚫 Conexión bloqueada
Revocación inmediata. El cliente descarga la CRL antes de cada conexión TLS y bloquea si encuentra el número de serie en la lista negra.
shell · revocación activa (crl) 
# Revocar un certificado inmediatamente (se añade a la CRL)
step ca revoke \
  --cert agent.crt \
  --key  agent.key \
  --reason "Clave comprometida" \
  --reasonCode 1
# reasonCode 1 = keyCompromise

# Si no tenés la clave (la perdiste): revocar con admin credentials
step ca revoke \
  --serial 4f3a2b1c... \
  --reason "Key lost" \
  --ca-url https://ca.internal:9000 \
  --admin-subject admin@internal

# Ver la CRL actual
step ca crl | step certificate inspect --bundle --insecure
